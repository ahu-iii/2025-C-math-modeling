第一部分

我们是怎么完成建模的

从 git 历史和文档结构反推，实际走的是"先定义、后计算"。这不是事后补的说法： 每次代码提交都跟着对应的 ADR 变更。

实线 = 主链路；绿线 = 用代码验证定义；橙色虚线 = 网格验证回路，下面第三条讲。 右下角的"论文初稿"是全图唯一靠人手连接的环节。

做得对的三件事

一、定义先于代码。 建模方案第二节先把"不适 / 适应期 / 缓解 / 复发 / 症状负担"定成可操作的判据， 再落到 ADR，最后才写代码。问题 2 和问题 3 的结论因此能对得上，不是各算各的。

二、否定掉的方案留了记录。 14 份 ADR 分"已定案""部分已弃用""待定"三种状态， 0007 的综合层就是被 0012 取代的。熵权法失效没有被悄悄删掉，而是留作敏感性对照，并在论文里写清楚失效原因： 同一个"缓解率"指标，在两种通行预处理下权重差了 423 倍（0.0011 vs 0.4532）。留着比藏起来划算。

三、把判断题改成了计算题。 这一招值得复用，但它现在只是 docs/建模待决问题.md 使用提醒里的一条脚注：

缓解判据、曾缓解判据、复发率分母这三条，原本当成"靠讨论共识解决、不靠算法解决"的判断题， 队伍一放假就卡住了。实际做法是把 2×3×2 = 12 组口径全部重跑一遍 （problem3_efficacy.py::definition_robustness()），结果 B 排第一在 12 组里全部成立， 权重全空间第一比例 97.9% 到 100.0%。换哪套口径只影响数值，不影响结论。

需要拍板的争议就此降成了需要说明的选择，最后按结构性理由挑定一套：两点判据与"豁免后持续不适"互为补集， r = −1.0000，会破坏 TOPSIS 的准则正交性。先跑网格再讨论，这个顺序应该写成默认协议， 别等下次卡住了才想起来。

第二部分 · 按严重度排序

5 处能提高产出效率与一致性的地方

  1. 01 要命 · 一致性

    口径没有唯一出处：缺一个 code/common.py

    现状

    建模方案.md 第八节执行清单第 1 条自己写着："敲定第二节的操作性定义，写入代码常量，避免口径不一致。" 要求提了，代码结构没跟上。三个脚本各写各的，唯一的共享方式是 P3 直接 import P2。

    证据
    • 同样三种药，代码里有 4 套标签字典：GROUP_LABELS（P1:42，"组1-药物C(对照)"）、 GROUPS（P2:54，"药物C(对照)"）、GROUP_SHORT（P2:363）、GROUP_LEGEND（P2:496）。P1 和 P2 的键值对不上。
    • onehot_to_category() 在 P1:91 和 P2:91 各写了一遍。P2 那份的 docstring 记了一个边界情形： "3/3149 受试者同一组内命中 >1 个 one-hot（录入笔误）"，P1 那份没记。 同一个数据坑，只有一半的代码知道。
    • setup_cjk_font() 复制了两份，字体候选列表也各写一遍。
    • P3 用 from problem2_factors import build_cohort, OUT_DIR, SUBST, TIMES 复用队列构造。 想法没错，但 P2 是个带副作用的可执行脚本：import 它会触发模块级的 setup_cjk_font()（P2:85）和 OUT_DIR.mkdir()（P2:52），顺带把 P2 的全部绘图代码拖进来。
    影响

    问题 4 是整合题，要同时吃问题 2 和问题 3 的结果，那就只能再 import 一次 P2，把这个畸形依赖坐实。 更现实的风险是：4 条待决问题一旦定案（比如抑郁程度改哑变量编码），改动点散在哪几个文件里，谁也说不准。

    怎么做

    抽一个 code/common.py：路径、药物标签（只留一套）、附件列名、字体、onehot_to_category、 load_baseline、load_followup、build_cohort， 还有 ADR 钉死的定义常量（D_t / 缓解 / 曾缓解 / 适应期 / 豁免后持续不适）。 P1 到 P4 一律 from common import ...，脚本之间互不 import。 这是纯机械重构，产出必须逐字节不变，正好拿现有的 24 个产物文件当回归基线。

  2. 02 要命 · 可复现

    没法一键重跑，也就没法确认产物是不是当前代码跑出来的

    现状

    没有 Makefile，也没有 run_all.py。24 个 CSV / PNG 提交进了 git，但仓库里没有任何东西 能回答"这些数字是不是今天这份代码跑出来的"。

    影响

    队友复工后 clone 下来，只能靠猜脚本的执行顺序，P3 依赖 P2 的模块这件事没有任何地方声明过。 更麻烦的是静默失配：改了 P2 的口径又忘了重跑 P3，problem3_topsis.csv 就还是旧的， 而论文引用的正是它，整个过程不会有任何报错。

    怎么做

    一个 Makefile（或 run_all.py）按 P1 → P2 → P3 → P4 全跑一遍，把依赖关系写成显式的。 再加一个 make check：跑到临时目录，然后 diff 已入库的产物，不一致就非零退出。 队友重跑之后要么逐字节相同，要么拿到一份清晰的 diff，这是保证一致性最省事的办法。

  3. 03 重要 · 效率

    论文里的每个数字都是手抄的

    现状

    论文初稿.md 有 1,112 行，里面的 OR=2.07、χ²=152.871、 p=0.0033、91.5%、r=−1.0000 全是从 output/*.csv 人工誊抄过来的常量。 上面那张图里，这是唯一一条没有代码连接的边。

    影响

    剩下 4 条待决问题（第 5、8、9、10 条）全部属于问题 2：抑郁程度用有序还是哑变量、 婚姻"未知"类别怎么处理、要不要补交互检验。任何一条定案，P2 的 OR 表都会变， 然后得有人在 1,112 行里手工找齐每一处引用。漏掉一处，论文里就有两个互相打架的数字， 而评委恰好会核对表格和正文。

    怎么做

    不用上模板引擎。最小可用的做法：每个脚本额外吐一份 output/*_claims.csv，记下可被论文引用的数字并配稳定 key； 再写一个 check_paper_numbers.py，把初稿里形如 OR=… / p=… / χ²=… 的数字全抓出来，逐个核对能不能在 claims 表里找到，对不上就报错并指出行号。 半小时的脚本，省掉每次改口径后的全文手工复查。

  4. 04 中等 · 一致性

    ADR 和代码之间只有注释，没有断言；总纲靠警告框打补丁

    现状

    代码注释大量引用 ADR（"见 ADR-0003""不可改用两点判据"），写得很好。但这层链接只是单向的散文： 谁要把 df["缓解"] = (df.D12 == 0) 改成两点判据，不会有任何东西失败。 ADR 不会变红，测试根本不存在，产物默默换一批新数字。

    证据
    • ADR 里已经写死了一批可以直接断言的事实：互补性 r = −1.0000、两率相加恒为 100.00%、 附件2 汇总栏 == 7 类症状之和（四时点零误差）、B 排第一 12/12。 文档已经把它们当事实陈述了，只是没写成 assert，改起来几乎不要成本。
    • 建模方案.md 第 5.3 节的正文现在是错的：它写着"熵权法（客观赋权）+ TOPSIS"， 靠上面压一个警告框写"本节已被 ADR-0012 取代、勿按本节原文实现"来纠正。 被取代的原文该收进 ADR-0007，那本来就是 ADR 的用途，而不是留在总纲正文里等人读到。
    怎么做

    写一个 code/test_definitions.py，把 ADR 声明的不变量逐条断言，十几行就够，不需要测试框架， 挂在 make check 里跑。这样 ADR 就不只是描述过去的决定，还能守住当下的代码。 另外把第 5.3 节按现口径重写，警告框里的决策背景移进 ADR-0007。

  5. 05 细微 · 协作

    协作约定写好了，但没人用

    现状

    CLAUDE.md 指向 docs/agents/issue-tracker.md 和 triage-labels.md， 规定了一整套 issue 驱动加分诊标签的协作约定。仓库里的实际状态是这样：

    约定文档要求仓库实际
    GitHub Issues任务的权威来源0 个（含已关闭）
    needs-triage规范标签未创建
    needs-info规范标签未创建
    ready-for-agent规范标签未创建
    ready-for-human规范标签未创建
    待决问题未规定躺在 markdown 里，共 4 条
    影响

    方案第九节把分工写得很清楚：非编程队友负责文献理由、结果解读和"模型建立"章节。 但 git 历史显示只有一条车道在跑，代码、文档、论文出自同一批提交。 4 条待决问题的"谁需拍板"栏全写着"全队"，而它们躺在一份 400 行的 markdown 里， 没有任何东西会去通知谁。队友复工第一件事是重读全文，而不是打开一个待办队列。

    怎么做

    把第 5、8、9、10 条各开一个 issue，打上 ready-for-human，标签先建出来。 权衡理由已经写好了，直接搬过去，每条 issue 就是一个读完就能拍板的单元。 顺带把第一部分那条网格协议写进 docs/agents/：卡住的判断题，先跑网格证明结论对口径不敏感，再挑一套并说明理由。 这一招在缓解判据、曾缓解判据、复发率分母上都用过，值得定成默认动作。

如果只做一件

先抽 common.py，再加 make check

01 和 02 得一起做：抽公共模块会动到全部三个脚本，而能证明重构没改数值的只有一件事，就是重跑产物逐字节相同。 所以先把校验建起来再动刀，拿现有的 24 个产物当基线。这两条做完，03、04、05 的成本都会降下来， 论文数字核对、ADR 断言、issue 队列都挂在同一条 make check 上。
