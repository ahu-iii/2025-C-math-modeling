# 网格协议（Grid Protocol）

一种把"口径之争"从议题降级为陈述句的方法。用于打破"需要开会拍板、但没人能开会"的僵局。

## 何时用

出现下面这种争议时：

- 有多种同样"合理"的口径/定义（缓解判据、分母选择、编码方式……），每种都能自圆其说；
- 团队直觉是"这个只能靠讨论/投票解决，算法算不出对错"；
- 但没人能马上拍板（放假、异步、时间紧张），争议卡在关键路径上。

不要在这种情况下干等团队开会。先跑网格，再讨论。

## 步骤

1. **枚举全部候选口径**，包括那些"显然不会选"的组合——不要预先筛选，网格协议的价值就在于覆盖全空间。
2. **对每种组合重跑下游结论**（不是重跑全部代码，只重跑受影响的那一段——通常是评估/排序/显著性检验）。
3. **看结论是否随口径变化**：
   - 如果核心结论（比如"哪个方案最优"）在**全部组合**下都不变——这个"争议"其实只改数值、不改结论，可以降级为"一句话说明用了哪个默认口径"，不必等团队拍板。
   - 如果结论随某些组合翻转——那才是真正需要人工判断的关键路径，标出翻转发生在哪个维度，把讨论范围缩小到那一个维度。
4. **在其中选一个具体口径时，优先用结构性理由**（例如：另一口径与已有指标数学等价/互补，会导致重复计数；或口径与题目原文的定义更吻合），而不是"图省事选了默认值"。把网格结果和结构性理由一起写进 ADR。

## 实战案例：`docs/建模待决问题.md` 第 1–3 条

三条口径争议——缓解判据（单点 vs 两点）、曾缓解判据（宽松 vs 严格）、复发率分母（曾缓解人数 vs 全体）——原本被登记为"需要全队开会拍板"，卡在问题3 的关键路径上，而团队当时在放假。

实际做法：把3个维度的口径组合全部枚举（2×3×2=12组），用 `code/problem3_efficacy.py::definition_robustness()` 重跑TOPSIS。两点无主诉分支先删除其互补指标，再做等权和权重扫描。结果为B在12/12组当前样本排序中都第一，固定样本权重扫描为95.5%–100.0%。

结论：三条争议会改变具体数值（例如代理缓解率从94.66%变到87.51%），但当前样本的排名方向不变。主口径仍应按数据含义与可复核性选择；如果使用两点无主诉口径，则必须删除与其互补的综合指标，避免TOPSIS重复计数。见ADR-0003、ADR-0004。
